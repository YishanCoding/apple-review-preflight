# Subscription — 订阅类 App 专项

> 本文件只覆盖**订阅类 App 的增量风险**。
> 通用检查项见 `all-apps.md`。
> IAP 技术细节见 `../guidelines/3-business.md`。

---

## 1. 订阅信息披露要求（3.1.1 / App Store 订阅规则）

### 1.1 价格披露

- [ ] **月度价格**必须在订阅选项界面上最显眼展示（即使用户选的是年度套餐）
- [ ] 年度套餐必须标注等效月均价：「¥168/年（约¥14/月）」
- [ ] 价格变更前必须通知存量订阅用户（StoreKit 会自动处理，但 Push 通知是加分项）

**为什么重要**：Apple 审核员专门检查年度套餐是否隐藏月度成本。常见拒审手法：「¥168/年」按钮旁边没有月均价，审核员认为用户不知道实际支出。

### 1.2 免费试用披露

- [ ] 试用结束后的收费金额和收费日期必须在订阅确认界面可见
- [ ] 文案示例：「7天免费试用，之后自动续订 ¥28/月，可随时取消」
- [ ] 不能仅在隐私政策或用户协议中说明试用条款

**为什么重要**：FTC 和 Apple 的「诱购」（dark patterns）打击重点。试用期结束扣款未告知是被拒高频场景。

---

## 2. 禁止「诱购」（Anti-Deceptive Patterns）的具体边界

以下行为均会触发 3.1.1 或 Apple 的额外审查：

| 违规行为 | 合规替代 |
|---------|---------|
| 关闭按钮极小或隐藏（Paywall 无法跳过） | 关闭/「不订阅」选项与购买按钮同等可见 |
| 「免费开始」按钮掩盖立即收费事实 | 按钮文案如实反映：「开始7天免费试用」 |
| 订阅确认后立即弹出升级套餐 | 仅在用户主动触发时展示升级选项 |
| 取消订阅入口深藏多层菜单 | 设置首页 → 订阅 → 管理（≤3步） |
| 取消流程中弹出「挽留折扣」后隐藏取消按钮 | 挽留界面仍保留直接取消选项 |

## 3. 订阅升降级的无缝体验

- [ ] 用户从低档升级到高档：立即生效，按差价比例收费（StoreKit 默认行为）
- [ ] 用户从高档降级到低档：当前周期结束后生效，界面上说明生效时间
- [ ] 跨订阅组的迁移：旧订阅取消 + 新订阅激活，需在界面上说明双重收费风险
- [ ] 升降级完成后在 App 内给用户明确状态确认（不只依赖系统邮件）

**为什么重要**：审核员会测试升降级流程。状态模糊（用户不知道是否已生效）是常见被拒点。

---

## 4. 转为订阅模式时的存量用户保护

适用场景：App 从「买断制」或「免费」改为订阅制。

- [ ] 存量用户（已购买旧版本）必须在合理时间内继续使用已付费功能
- [ ] 不能通过订阅墙直接锁住存量用户已付费解锁的功能
- [ ] Review Notes 中说明：存量用户政策、过渡方案、时间线

**为什么重要**：Apple 2.1(b) 要求 IAP 功能在审核期间可访问。存量用户功能被锁会被视为「破坏性更新」，有被拒风险。历史上多个知名 App 因此被拒并登上科技媒体。

---

## 5. 自动续订文案要求

StoreKit 标准弹窗会显示续订条款，但 App 内的自定义文案也必须合规：

**必须包含**（推荐用 `SubscriptionStoreView` 或自定义 UI 时）：
- 订阅名称
- 价格和计费周期
- 免费试用期（如有）
- 「订阅自动续订，可随时在设置中取消」

**禁止**：
- 仅说「开始使用」而不提续订
- 用「激活」「解锁」等词掩盖付费事实

## 6. 消耗品 Credits 在订阅中的使用边界

适用场景：订阅套餐中包含每月 Credits（如 AI 调用次数、生成次数）。

- [ ] Credits 是否跨月累积必须在界面上明确说明
- [ ] Credits 不累积时，审核员不会因此拒审，但用户看不懂会投诉，影响 App 评分
- [ ] 订阅中的 Credits 属于消耗品，不适用非消耗品的「恢复购买」逻辑
- [ ] 如果 Credits 可独立购买（不依赖订阅），需在 App Store Connect 中单独创建消耗品 IAP

**为什么重要**：混淆消耗品/非消耗品/订阅的关系会在 3.1.1 审查时被追问。

## 7. 常见被拒场景和修复

### 场景 A：Paywall 无法关闭
**被拒理由**：App 首启动后直接全屏 Paywall，无关闭按钮。
**修复**：右上角加「×」或「跳过」；如必须展示，至少 3 秒后出现关闭按钮。

### 场景 B：订阅期间 Web 端变更导致 App 功能消失
**被拒理由**：订阅用户打开 App 发现功能不见了，审核员会认为 IAP 功能不可访问（2.1b）。
**修复**：确保 App 内已解锁功能不因 Web 端配置变更而消失；审核期间后端配置冻结。

### 场景 C：恢复购买按钮缺失
**被拒理由**：IAP 界面无「恢复购买」选项。
**修复**：在 Paywall 底部添加「已有订阅？恢复购买」文字按钮；触发 `Transaction.currentEntitlements` 检查。

### 场景 D：价格显示不一致
**被拒理由**：App 内显示 ¥28/月，但 App Store 产品页显示不同价格。
**修复**：价格动态从 StoreKit 读取，不要硬编码。使用 `Product.localizedPrice`。

### 场景 E：审核期间订阅无法购买
**被拒理由**：沙盒环境未配置，或后端对沙盒账号有白名单限制。
**修复**：在 Xcode scheme 中开启 StoreKit 测试模式；解除后端对沙盒账号的限制。
